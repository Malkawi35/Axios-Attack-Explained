01 — Timeline

How It Unfolded (UTC)

The entire attack was planned, staged, and executed in under 24 hours — targeting a Sunday night maintenance window to maximize dwell time before detection.

Mar 30 — ~06:00 UTC (T-18h)
Staging: plain-crypto-js@4.2.0 published
Attacker creates a clean decoy package mimicking the legitimate crypto-js library. Version 4.2.0 is benign — designed to build npm registry history and avoid suspicion.
Mar 31 — 00:21 UTC
Compromise: axios@1.14.1 published
Using a stolen long-lived npm token, the attacker hijacks the maintainer account, changes the email to a ProtonMail address, and publishes a malicious release tagged "latest." Bypasses all CI/CD pipelines.
Mar 31 — ~00:30 UTC
Payload: plain-crypto-js@4.2.1 goes live
The weaponized dependency is published — contains the obfuscated dropper (setup.js) that triggers via npm's postinstall hook on every npm install.
Mar 31 — 01:00 UTC
Legacy branch: axios@0.30.4 published
Attacker also poisons the "legacy" dist-tag, maximizing blast radius across projects on both current and legacy Axios APIs.
Mar 31 — ~01:50 UTC
Detection: Security researchers raise alarm
StepSecurity, Elastic, and Huntress detect anomalous C2 traffic. Elastic files a GitHub Security Advisory. Community alerts begin spreading on social media.
Mar 31 — 03:15 UTC
Takedown: npm pulls malicious packages
Both poisoned Axios releases and plain-crypto-js are removed from the registry. Safe versions (1.14.0 and 0.30.3) are restored as latest. C2 infrastructure goes offline.
02 — Attack Chain

Step-by-Step Kill Chain

Click play to watch the attack unfold stage by stage. This is the exact technical sequence from account takeover to RAT deployment.

🔑
1. Credential Theft
Attacker obtains a long-lived npm access token for the jasonsaayman maintainer account. Changes associated email to ifstap@proton.me to lock out the real owner.
📦
2. Malicious Dependency Injection
A new dependency plain-crypto-js@^4.2.1 is added to package.json. It's never imported in Axios source — it only exists to trigger its postinstall hook.
"dependencies": { "plain-crypto-js": "^4.2.1" }
⚡
3. Postinstall Hook Fires
When anyone runs npm install, npm automatically executes setup.js from plain-crypto-js. No user interaction needed — just installing triggers the payload.
"scripts": { "postinstall": "node setup.js" }
🔍
4. OS Detection (SILKBELL)
The obfuscated dropper (setup.js / SILKBELL) checks process.platform to determine if the target is Windows, macOS, or Linux — then downloads the matching stage-2 binary from sfrclak[.]com:8000.
🦠
5. RAT Deployment (WAVESHAPER.V2)
Platform-specific RAT binaries are dropped: wt.exe (Windows), macWebT (macOS), or Linux variant. All share identical C2 protocol, command structure, and beacon behavior — a unified cross-platform implant.
📡
6. C2 Beacon & Exfiltration
The RAT beacons to 142.11.206.73 using POST requests disguised as npm registry traffic (to packages.npm.org — actually a domain owned by the National Association of Pastoral Musicians since 1997). Supports remote shell, file upload/download, and credential harvesting.
🧹
7. Anti-Forensics Cleanup
After execution, the dropper replaces plain-crypto-js/package.json with a clean version — no postinstall script, no setup.js. Running npm audit afterward shows nothing. The compromise is invisible.

04 — AI in Modern Attacks

How Hackers Leverage AI & Automation

This attack specifically is attributed to North Korean state actors (UNC1069 / BlueNoroff). But here's how AI and automated tooling are accelerating supply chain attacks across the board in 2026.

Malware like QUIETVAULT uses AI/ML analysis to discover secrets on compromised systems — distinguishing test credentials from production keys, prioritizing high-value targets for deeper penetration. Context-aware exfiltration, not just blind scraping.

⚡ AI Vulnerability Scanners RECON

Attackers use AI to rapidly scan public-facing applications for misconfigurations and missing auth controls. IBM's 2026 X-Force Index reported a 44% increase in attacks starting with exploited public apps, driven partly by AI-enabled vulnerability discovery.

05 — Defense

How to Protect Yourself

🔒 Pin Dependencies

Never use floating version ranges like ^1.14.0. Pin exact versions in lockfiles. The attack only worked because npm install auto-resolved to the malicious latest.

🚫 Disable postinstall

Use --ignore-scripts in CI/CD, or switch to pnpm which blocks lifecycle scripts by default. This single setting would have prevented the dropper from executing.

🔐 Short-Lived Tokens

The attacker used a long-lived npm access token. Migrate to short-lived, scoped automation tokens and require OIDC provenance for publishing.

✅ Verify Provenance

Legitimate Axios releases include SLSA build attestations linking to a specific GitHub Actions run. The malicious versions had none — absence of provenance should trigger alerts.

🛡️ Supply Chain Firewalls

Tools like Datadog's SCFW, Socket, or Snyk wrap npm install to automatically block known malicious packages before they execute.

🔄 Rotate Everything

If exposed: npm tokens, AWS keys, SSH keys, cloud creds, CI/CD secrets, and .env values. The RAT supported full credential harvesting — assume everything was exfiltrated.
